## The business problem

A company runs its core transactional system on an **on-premises SQL Server** and
wants a continuously updated copy in **Amazon RDS for SQL Server** — to feed cloud
analytics and reporting, to enable a phased migration, or to keep a warm DR replica
in the cloud. The requirements are the awkward part: it has to be **near-real-time**
(no nightly ETL window), run **without downtime** on the source, and the replica
must keep the **exact same primary keys** — which are `IDENTITY` columns that SQL
Server normally won't let you insert into.

The shape of the solution: **Debezium** captures every change from the on-prem SQL
Server via CDC and streams it through Kafka (here, **Amazon MSK**) over a private
**AWS Direct Connect / VPN** link; the **Debezium JDBC sink** then writes those
changes into RDS — preserving the identity values natively, with no custom code.

## Try it locally

The [repo](https://github.com/ohad-israeli/debezium-jdbc-sqlserver-sink) is a
self-contained stand-in for that architecture. Instead of real on-prem hardware and
an RDS instance, it runs **two SQL Server containers** (source + target) wired up
with the same Debezium CDC source and JDBC sink, so you can watch the whole flow on
a laptop with `docker compose`. Pointing it at a real on-premises source and an RDS
endpoint later is mostly a matter of connection strings.

The source table is seeded with deliberately non-contiguous identity values
(`1, 11, 21, 31, 41`) so it's obvious whether the *exact* keys survived the trip
rather than getting renumbered by the target.

## The key piece

Everything hinges on one line in the sink config. The sink runs in `upsert` mode
keyed on the Debezium record key, and turns on SQL Server identity insert:

```json
{
  "connector.class": "io.debezium.connector.jdbc.JdbcSinkConnector",
  "topics": "sqlserver_source.SourceDB.dbo.TBL_AG_TEST4",
  "insert.mode": "upsert",
  "delete.enabled": "true",
  "primary.key.mode": "record_key",
  "primary.key.fields": "col1",
  "table.name.format": "TBL_AG_TEST4",
  "dialect.sqlserver.identity.insert": "true"
}
```

With `dialect.sqlserver.identity.insert` set to `true`, the sink wraps each write
batch in `SET IDENTITY_INSERT <table> ON` / `OFF`, so the source's identity values
are inserted verbatim instead of being rejected or re-generated. This needs
Debezium JDBC connector **v3.0.7 or newer**.

## Running it

```bash
git clone https://github.com/ohad-israeli/debezium-jdbc-sqlserver-sink
cd debezium-jdbc-sqlserver-sink
./scripts/up.sh
```

`up.sh` brings up the stack, enables CDC and seeds the source, registers both
connectors, and prints the source and target side by side. After it settles, the
target table holds the same five rows with the same `col1` values. Insert another
row on the source and re-run `scripts/verify.sh` to watch it flow through.

> Heads up: this runs **two** SQL Server containers plus Kafka, Connect, and
> Schema Registry — give Docker ~8 GB of RAM.

## Why it matters

On-prem-to-cloud database replication is usually where teams reach for a managed
service or a pile of bespoke scripts. The Debezium CDC source into the Debezium
JDBC sink is a clean, open-source path that handles it as a stream — and SQL Server
identity columns, which used to force you onto a custom sink connector, are now a
single config flag. That makes the off-the-shelf connectors a viable answer for a
lot more replication and migration jobs.

Full code, configs, and SQL are in the [repo](https://github.com/ohad-israeli/debezium-jdbc-sqlserver-sink).
