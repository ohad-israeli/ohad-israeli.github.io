## The business problem

A company runs its core transactional system on an **on-premises SQL Server** and
needs a continuously updated copy in **Amazon RDS for SQL Server** — to power cloud
analytics and reporting, support a phased migration to the cloud, and keep a warm
disaster-recovery replica. That copy has to stay current in **near-real-time**, put
**no downtime or load on the production source**, and reproduce the data **exactly** —
right down to the `IDENTITY` primary keys that downstream systems rely on.

The shape of the solution: **Debezium** captures every change from the on-prem SQL
Server via CDC and streams it through **Confluent Cloud** — managed Kafka and Schema
Registry, with the Confluent Cloud Console as the UI — and the **Debezium JDBC sink**
writes those changes into RDS over **AWS Direct Connect**, preserving the identity
values natively, with no custom code.

## Try it yourself

The [repo](https://github.com/ohad-israeli/debezium-jdbc-sqlserver-sink) is a
runnable version of this architecture. Kafka, Schema Registry, and the UI are real
**Confluent Cloud**; the only things that run locally (via `docker compose`) are the
self-managed **Kafka Connect** worker hosting the two Debezium connectors, and **two
SQL Server containers** that stand in for the on-prem source and the RDS target — so
you can exercise the whole pipeline from a laptop. Pointing it at a real on-prem
database and an RDS endpoint later is mostly connection strings.

The source table is seeded with deliberately non-contiguous identity values
(`1, 11, 21, 31, 41`) so it's obvious whether the *exact* keys survived the trip
rather than getting renumbered by the target.

## The key piece

Everything hinges on one line in the sink config. The sink runs in `upsert` mode
keyed on the Debezium record key, and turns on SQL Server identity insert:

```json
{
  "connector.class": "io.debezium.connector.jdbc.JdbcSinkConnector",
  "topics": "sqlserver_source.SourceDB.dbo.TBL_AG_TEST4",
  "insert.mode": "upsert",
  "delete.enabled": "true",
  "primary.key.mode": "record_key",
  "primary.key.fields": "col1",
  "table.name.format": "TBL_AG_TEST4",
  "dialect.sqlserver.identity.insert": "true"
}
```

With `dialect.sqlserver.identity.insert` set to `true`, the sink wraps each write
batch in `SET IDENTITY_INSERT <table> ON` / `OFF`, so the source's identity values
are inserted verbatim instead of being rejected or re-generated. This needs the
Debezium JDBC connector **v3.0.7 or newer**.

## Running it

```bash
git clone https://github.com/ohad-israeli/debezium-jdbc-sqlserver-sink
cd debezium-jdbc-sqlserver-sink
cp .env.example .env      # fill in your Confluent Cloud bootstrap + API keys
./scripts/up.sh
```

`up.sh` first runs a **preflight** that validates `.env` and checks that it can
reach Confluent Cloud (Schema Registry auth + broker connectivity), then brings up
Connect and the two SQL Servers, seeds the source, registers both connectors, and
prints the source vs. target rows. After it settles, the target holds the same five
rows with the same `col1` values. Insert another row on the source and re-run
`scripts/verify.sh` to watch it flow through; inspect the topics and schemas in the
Confluent Cloud Console.

> You'll need a Confluent Cloud cluster with Schema Registry enabled, and ~6 GB of
> RAM free for the two SQL Server containers plus Connect.

## Why it matters

On-prem-to-cloud database replication is usually where teams reach for a managed
service or a pile of bespoke scripts. Debezium CDC into the Debezium JDBC sink, with
the streaming backbone on Confluent Cloud, is a clean path that handles it as a
stream — and SQL Server identity columns, which used to force you onto a custom sink
connector, are now a single config flag. That makes the off-the-shelf connectors a
viable answer for a lot more replication and migration jobs.

Full code, configs, and SQL are in the [repo](https://github.com/ohad-israeli/debezium-jdbc-sqlserver-sink).
